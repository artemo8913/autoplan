# Промт для Claude Code: АС План КС

## Контекст проекта

Ты помогаешь разрабатывать web-приложение **«АС План КС»** — генератор планов контактной сети и устройств электроснабжения для железных дорог.

### Стек
- React 18+ / Vite / TypeScript
- MobX (пока минимально, в перспективе — глобальный store)
- SVG для отрисовки (не Canvas)
- Без бэкенда (данные в памяти, позже — экспорт/импорт JSON)

### Текущее состояние кода
Проект уже содержит работающий прототип с:
- `Railway` — класс оси дороги (генерирует линейные координаты)
- `Track` — класс пути (смещение от оси, направление odd/even)
- `Pole` — класс опоры (позиционирование по габариту от пути)
- `Attachment` — связь опора↔путь (геометрически вычисляет черту-консоль)
- SVG-компоненты: `TrackFigure`, `PoleFigure`, `AttachmentFigure` + соответствующие Layer-компоненты
- Pan/zoom на SVG

### Ключевые документы в проекте
1. **`docs/gost-reference.md`** — справочник условных обозначений планов КС по Порядку ОАО «РЖД» 2023. Содержит описания всех символов (опоры, подвески, провода, сопряжения, разъединители и т.д.) со ссылками на рисунки документа. **Обязательно прочитай этот файл перед началом работы.**
2. **`src/components/gost-symbols.tsx`** — готовые SVG React-компоненты условных обозначений (PoleBase, ConsoleSymbol, GuyWireSymbol, CatenarySymbol, ZigzagSymbol, MidAnchorSymbol, и т.д. — ~20 компонентов). **Прочитай этот файл — компоненты нужно использовать, а не писать с нуля.но не факт, что они сгенерированный правильно. я их буду вручную переделывать в каждый отдельный файл**
3. **`src/types.ts`** — базовые типы: Pos, Poses, RelativeSidePosition и т.д.

### Архитектурные принципы (уже согласованы)
- Кривые не реализуем — пути всегда прямые (по п. 2.12 Порядка)
- `Attachment` — это визуальная связь «опора → путь» (консоль на плане). Направление черты вычисляется геометрически: `Math.sign(trackY - poleY)`. Не нужно хранить «лево/право» для отрисовки.
- `startPos` / `endPos` у Attachment — геттеры, а не поля. Вычисляются из актуальных позиций опоры и пути.
- Позиции в `Track` пока хранятся как словарь `Poses` (ключ = метр). Это неоптимально, но работает. Переход на `getPositionAtX(x)` — позже.
- ID генерируются через `crypto.randomUUID()`
- Будет реализована структура по системе FSD (Feature Sliced Design)

## Принцип работы

**Разработка ведётся поэтапно и итерационно.**

Перед каждым этапом:
1. Кратко опиши, что будешь делать (2-3 предложения)
2. Дождись моего подтверждения или корректировки
3. Реализуй **минимальный** работающий шаг (как правило, один файл или одно небольшое изменение)
4. Покажи результат, дождись обратной связи
5. Только потом переходи к следующему шагу

**НЕ ДЕЛАЙ:** не выгружай сотни (тысячи) строк кода за раз. Каждый коммит — маленький, понятный, тестируемый.

**ДЕЛАЙ:** объясняй решения, предлагай варианты, спрашивай если неоднозначно.

## План этапов

### Этап 1. Интеграция GOST-символов в существующую отрисовку
**Цель:** обновить текущий примитивный `PoleFigure` (просто `<circle>`) на GOST-корректные символы из `gost-symbols.tsx`.

Шаги:
1. Добавить поле `material` в класс `Pole` (concrete / metal)
2. Переписать `PoleFigure` — использовать `PoleBase` из gost-symbols
3. Переписать `AttachmentFigure` — использовать `ConsoleSymbol` из gost-symbols
4. Добавить `PicquetSymbol` для отрисовки пикетных меток на оси
5. Проверить визуально, что план выглядит корректно

### Этап 2. Контактная подвеска (провода вдоль путей)
**Цель:** нарисовать линии КС от опоры к опоре вдоль пути.

Шаги:
1. Определить класс/интерфейс `AnchorSection` (анкерный участок: начальная опора, конечная опора, точки фиксации, тип подвески)
2. Создать компонент `CatenaryFigure` — рисует рабочую подвеску вдоль пути между опорами анкерного участка
3. Создать `CatenaryLayer`
4. Добавить тестовые данные — пару анкерных участков
5. Добавить `AnchorSectionArrow` (стрелка с длиной и номером АУ)

### Этап 3. Дополнительные провода (ДПР, питающие, ВЛ)
**Цель:** нарисовать прочие провода с правильными dash-паттернами.

Шаги:
1. Определить интерфейс `WireLine` (тип провода, точки подвеса?, начало, конец)
2. Компонент `WireFigure` — использует `getWireDashPattern()` из gost-symbols
3. Добавить маркировочные подписи (повторяются через 2-3 пролёта)

### Этап 4. Анкерные опоры, оттяжки, фиксирующие опоры
**Цель:** визуально различать типы опор.

Шаги:
1. Добавить поле `poleType` в `Pole` (intermediate / anchor / fixing)
2. В `PoleFigure` — отрисовка `GuyWireSymbol` для анкерных, `FixingCrossSymbol` для фиксирующих
3. Синий цвет для анкерных опор ИС

### Этап 5. Зигзаги и нумерация
**Цель:** добавить треугольники зигзагов, номера опор, длины пролётов.

Шаги:
1. Добавить данные зигзагов (направление, тип) к связи опора↔подвеска
2. `ZigzagSymbol` на каждой опоре
3. `PoleNumberLabel` рядом с каждой опорой
4. `SpanLengthLabel` между опорами

### Этап 6. Сопряжения анкерных участков
**Цель:** показать неизолирующие и изолирующие сопряжения.

Шаги:
1. Определить данные сопряжения (тип, опоры, секции)
2. Отрисовка: два участка «внахлёст», нерабочая ветвь штриховой линией
3. Обозначение изоляции (для ИС)

### Этап 7. Средние анкеровки
**Цель:** символ средней анкеровки на соответствующих опорах.

Шаг: `MidAnchorSymbol` на нужных опорах.

### Этап 8. MobX Store и интерактивность
**Цель:** глобальное хранилище и возможность редактирования.

Шаги:
1. `ProjectStore` — хранит Railway, Track[], Pole[], AnchorSection[]
2. Все позиции — computed из store
3. Клик по опоре → панель свойств
4. Редактирование типа, материала, габарита
5. Добавление / удаление опор

### Этап 9. Таблица-сетка и спецификация
**Цель:** таблица-сетка внизу плана (элемент плана, профиля, ординаты) + спецификация.
