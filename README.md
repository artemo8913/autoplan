# React + TypeScript + Vite

This template provides a minimal setup to get React working in Vite with HMR and some ESLint rules.

Currently, two official plugins are available:

- [@vitejs/plugin-react](https://github.com/vitejs/vite-plugin-react/blob/main/packages/plugin-react) uses [Babel](https://babeljs.io/) (or [oxc](https://oxc.rs) when used in [rolldown-vite](https://vite.dev/guide/rolldown)) for Fast Refresh
- [@vitejs/plugin-react-swc](https://github.com/vitejs/vite-plugin-react/blob/main/packages/plugin-react-swc) uses [SWC](https://swc.rs/) for Fast Refresh

## React Compiler

The React Compiler is not enabled on this template because of its impact on dev & build performances. To add it, see [this documentation](https://react.dev/learn/react-compiler/installation).

## Вопросы

1. Анкер отдельной сущностью сделать и отрисовывать как отдельный элемент? По идее, анкер неотъемлимая часть анкерной опоры - если опора анкерная, то можно скакзать 100% у неё есть анкер (свойство опоры, если анкерная, то рисовать анкер). С другой стороны, анкер может быть старый, не демонтированный и не подцеплен к опоре (анкер - отдельный элемент, если есть оттяжка, то рисуется линия или две от анкера до опоры). Пока делаю, как свойство опоры
2. `Pole.ts` - все свойства, связанные с анкером вынести в одно поле. Сделать класс `Anchor`. Поля в последствии сделать приватными.
3. Оптимизация перерисовок. Сейчас любой чих приводит к перерисовке всего и вся. Это неприемлимо. Хочется, чтобы менялись непосредственно те элементы, данные которых изменились.

## UI/UX эксперименты для сравнения

### Эксперимент 1: Inline-редактирование vs боковая панель

Текущая боль: клик по опоре → перевести мышь к панели → клик по полю → редактирование. Три подхода для сравнения:

- **A) Floating micro-panel:** Двойной клик на опору → маленькая floating-панель рядом с опорой (имя + габарит + зигзаг). Одиночный клик — полная панель справа.
- **B) In-place SVG editing:** Двойной клик на текст имени → текст становится editable input через `foreignObject` прямо в SVG.
- **C) Hybrid (рекомендация):** Одиночный клик = выделение + боковая панель. Двойной клик на конкретный визуальный элемент (имя, зигзаг, длина пролёта) → single-field inline editor через абсолютно позиционированный HTML-блок (`getBoundingClientRect()`).

**Рекомендация: C** — уважает существующую архитектуру, добавляет точечное редактирование.

### Эксперимент 2: Drag-and-drop

- **A) Immediate drag:** Клик на выделенную опору + тяг — сразу перемещает. Риск: случайные перетаскивания.
- **B) Drag threshold (рекомендация):** Уже есть `DRAG_THRESHOLD = 4` в InputHandler. Если мышь сдвинулась > 4px при удержании на выделенной сущности → режим drag. Стандартный подход.
- **C) Move-tool:** Отдельный инструмент "Перемещение" в toolbar. Явный intent, но лишний клик.

**Рекомендация: B** — инфраструктура уже есть в `InputHandler._updateDragThreshold`.

### Эксперимент 3: Создание АУ / ИС

- **A) Wizard:** Кнопка toolbar "Создать АУ" → клик на первую граничную опору → клик на последнюю → модалка подтверждения.
- **B) Selection-based:** Выделить опоры лассо → ПКМ → "Создать АУ из выделенного". Опоры сортируются по X, первая и последняя — границы.
- **C) Table-driven:** Отдельная панель "Анкерные участки" со списком. Кнопка "Новый" → модалка с выбором start/end опор.

**Рекомендация: реализовать A и B** — они покрывают разные workflow (точное размещение vs быстрое создание).

### Эксперимент 4: Редактирование длины пролёта

- **A) Click label (рекомендация):** Клик на число длины → inline NumberInput. Shift = сдвинуть цепочку, без = только одну опору.
- **B) Drag handle:** Drag-ручка на лейбле пролёта. Тянешь влево/вправо — расстояние меняется с live-preview.
- **C) Table:** Длины пролётов редактируются в таблице Stage 9.

**Рекомендация: A** для быстрых правок, **C** как production-подход в Stage 9.
